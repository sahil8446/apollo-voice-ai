# Apollo Voice AI — Appointment Receptionist

A voice AI receptionist for **Apollo Spectra Hospitals, Pune** that books,
reschedules, cancels, and looks up doctor appointments autonomously over a
phone or web call. Built on **Retell** (voice) + **FastAPI** + **PostgreSQL**,
with a re-runnable evaluation harness and a read-only admin dashboard.

> **Status:** backend verified end-to-end against PostgreSQL — 22/22 unit/API
> tests pass, 7/7 eval scenarios pass, and a concurrency proof confirms no
> double-booking under 10 simultaneous bookings. See [Verification](#verification).

## Architecture in one paragraph

Retell handles speech and the LLM, and calls six tools. Each tool POSTs
`{name, args, call}` to a FastAPI endpoint under `/retell/*`, which delegates to
a shared service layer backed by PostgreSQL — the single source of truth. The
service tier is **stateless** (scale out by running more processes) and the
booking path is **double-booking-proof** via a row lock plus a partial unique
index. A clean REST API (`/doctors`, `/availability`, `/appointments`) exposes
the same logic for the eval harness and admin UI. Full detail in
[`docs/architecture.md`](docs/architecture.md).

### Why these choices
- **Retell** — lower, more consistent latency (~600–800 ms) and built-in LLM
  simulation testing; purpose-built for scheduling/IVR.
- **FastAPI (async)** — clean OpenAPI that maps 1:1 to Retell tools; async fits
  an I/O-bound, DB-backed workload.
- **PostgreSQL** — relational integrity is exactly what slot/appointment
  conflict checks need, and partial unique indexes give a hard anti-double-book
  guarantee no application code can violate.
- **Alembic** — real products evolve their schema without wiping data.

## Quick start

### Option A — Docker (one command)
```bash
docker compose up --build
```
Brings up PostgreSQL + the API, runs migrations + seed automatically, and serves:
- API: <http://localhost:8000>  · OpenAPI docs: <http://localhost:8000/docs>
- Admin dashboard: <http://localhost:8000/>  (key: `dev-admin-key-change-me`)

### Option B — Local Python
```bash
cd backend
python -m venv .venv && source .venv/Scripts/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env          # defaults point at a local Postgres

# with a Postgres running (e.g. docker run -p 5432:5432 ... postgres:16):
alembic upgrade head          # create schema
python -m app.seed.seed       # load Apollo data + generate slots
uvicorn app.main:app --reload
```

## Run the eval harness
```bash
cd eval
python run_eval.py --base-url http://localhost:8000
```
Runs the scenario suite against the live API, writes metrics to
`eval/results/`, prints a pass/fail summary, and exits non-zero on failure (CI-ready).

**Dimensions measured & why** (a voice booker is judged on whether the caller
leaves with the right appointment and never hits dead air):

| Dimension | What it tells you |
|---|---|
| **task_success** | Did the action actually complete (book / reschedule / cancel)? |
| **tool_correctness** | Right endpoint, valid params, expected status/body vs. real DB state |
| **conflict_handling** | When a slot is taken, are valid alternatives offered (never a dead-end)? |
| **robustness** | Vague input → department, mid-call change, bad ids, empty lookups |
| **latency** | Per-call round-trip; reported as p50/p95 (the voice SLO) |

**Scenarios:** happy-path booking · booking into a taken slot (conflict) ·
vague symptom → department · mid-call doctor change · reschedule · cancel ·
nonexistent appointment.

**Latest committed result** (`eval/results/latest.json`): **7/7 scenarios pass**,
backend latency **p50 ≈ 10 ms, p95 ≈ 25 ms**.

## Wiring up Retell
1. Deploy the backend (below) and note its public URL.
2. In `retell/tools.json`, replace `{{BASE_URL}}` with that URL.
3. Create a Retell agent, paste `retell/prompt.md` as the system prompt, and add
   the six custom functions from `tools.json`.
4. Set `RETELL_API_KEY` in the backend env and `VERIFY_RETELL_SIGNATURE=true`.
5. Assign a phone number (or use Retell's web test widget) and call.

The six tools — `find_doctor`, `check_availability`, `book_appointment`,
`reschedule_appointment`, `cancel_appointment`, `lookup_appointments` — each map
to one `/retell/*` endpoint.

## Deploy (Render)
```bash
# Connect the repo in Render, or:
render blueprint launch        # uses render.yaml
```
`render.yaml` provisions a free Postgres + the Docker web service, wires
`DATABASE_URL`, generates a strong `ADMIN_API_KEY`, and turns on Retell
signature verification. The container entrypoint runs `alembic upgrade head` and
the idempotent seed on every deploy. Railway works the same way (it injects
`DATABASE_URL`, which the app auto-normalizes to the asyncpg driver).

## Known limitations
- **Slots are seeded, not synced** from a real HIS/EMR — OPD windows are
  realistic but assigned (exact public times aren't published).
- **The eval harness tests the backend contract**, not STT/TTS/LLM behaviour —
  that lives in Retell's own simulation suite. End-to-end voice quality is
  validated by live calls (see the Loom).
- **Single-tenant** — schema is clean for multi-clinic (`clinic_id` + RLS) but
  that isn't built.
- **Reschedule stays with the same doctor** by design; cross-doctor changes are
  a cancel + rebook.
- **Cache is in-process** — correct for a single instance and fine across
  replicas (TTL-bounded staleness); swap to Redis if you need cross-node
  invalidation.
